# init — platform bring-up

The bring-up sequence that turns a bare ESP-IDF `app_main` into a running
spangap device: it mounts the filesystems, loads the config tree, starts the
foundation tasks (log, CLI, power management, auth), and walks the
build-generated service registry that brings up every other straddle in the
build. Header: [`spangap.h`](../esp-idf/include/spangap.h),
[`service.h`](../esp-idf/include/service.h).

## How a device boots

A spangap firmware's `app_main` is **generated by the build**, not hand-written.
Every straddle's boot participation is a **`Service`**
([service.h](../esp-idf/include/service.h)) — a platform object with a boot
lifecycle — and `app_main` constructs them all, then drives the boot phases by
walking them:

```cpp
extern "C" void app_main() {
    spangapRegisterServices();     // construct + register every staged straddle's Service, in boot order
    serviceRunStart();             // onStart walk: bare-hardware band (board HAL pre-init)
    spangapInit();                 // core foundations (fs, storage, log, cli, pm, auth)
    spangapSettingsGenDefaults();  // declarative settings: seed storage defaults
    serviceRunInit();              // onInit walk: every service, with the ecosystem up
    spangapSettingsGenRegister();  // declarative settings: register LCD panes (if spangap-lcd staged)
    spangapPostAppInit();          // finalise: boot script, sys.boot_complete, first cron poll
}
```

You do not write this and you do not construct or init any straddle's objects by
hand. A straddle that needs to run code at boot declares a **`services:`** entry
(a `Service` subclass) in its `straddle.yaml`; `spangap-inside` generates the
code that constructs it and appends it to the ordered registry when that straddle
is staged. Legacy `start:`/`init:` free-function hooks still work — the generator
wraps each in an adapter `Service` — but `services:` is the modern form. **If a
straddle is in the build, it comes up automatically.**

### The model

`spangapRegisterServices()` constructs every service and appends it to one
ordered registry (`serviceRegister`). **Registration order is boot order:** the
generator emits the registrations in `init_order()` — the platform band (core,
net, web, lcd) first, then the dependency-topological straddle band — so a
service's dependencies are always registered, and brought up, before it.
`app_main` then walks that one registry twice, and a service takes part in a
phase purely by overriding that phase's virtual:

- **`serviceRunStart()` → `onStart()`** — the **bare-hardware** band, before
  `spangapInit()`. For board bring-up the platform itself depends on: powering a
  peripheral rail before the SD bus is touched, registering the display/touch HAL
  before the LCD straddle wants it. An `onStart` runs with **no** `info()`,
  storage, `fs_*`, or ITS available — raw peripherals only.

- **`spangapInit()`** — brings up the core foundations only: filesystem mounts,
  the active state-store choice, the config tree, then the log / CLI / power /
  auth tasks. It returns with a usable platform but **no siblings yet**. The
  storage *task* and cron are ordinary services like everything else — they come
  up in the `onInit` walk, not here.

- **`serviceRunInit()` → `onInit()`** — the ordinary bring-up band, after
  `spangapInit()`. A service's `onInit` can count on storage/cron, the IP stack,
  the web stack, and (by its band position) the LCD shell already being up, so it
  neither defers nor orders itself against them.

- **`spangapPostAppInit()`** — finalises boot after everything is up: marks RTC
  RAM valid, runs the `boot` CLI script, publishes `sys.boot_complete`, applies
  log levels, and fires the first cron poll. IDF's `main_task` deletes itself
  after this returns.

**Ecosystem-free constructors.** Every service is constructed at the very top of
`app_main` — before `serviceRunStart` and `spangapInit` — so a `Service`
constructor must do **member init only**: heap/PSRAM are up, but
storage/fs/log/cli/ITS are not. All real work goes in `onStart`/`onInit`.

## Boot barriers — the part consumers do use

A straddle that must wait for a runtime condition before doing its work uses the
two barrier primitives in [`spangap.h`](../esp-idf/include/spangap.h):

```cpp
// Hold until the network straddle has signalled the mesh is ready (or 120 s).
waitForFlag("rns.ready", 120);

// Hold until the clock is known-valid before stamping anything with a timestamp.
waitForTime(0);   // 0 → the operator-tunable s.sys.time_wait_s default
```

- **`waitForTime(timeout_s)`** blocks until `sys.time.valid` flips to `1` (a time
  source — SNTP in spangap-net, GPS/RTC in a board straddle — has synced), or the
  timeout elapses. `timeout_s <= 0` uses `s.sys.time_wait_s` (fallback 30 s); set
  that key to `0` on an offline node with no time source to skip the wait.
- **`waitForFlag(key, timeout_s)`** blocks until any ephemeral storage int `key`
  is non-zero. This is how interfaces and clients gate on readiness flags another
  straddle publishes (`rns.ready`, `net.up`, …). `timeout_s <= 0` checks once
  without waiting.

Both hold a shared power-management no-deep-sleep lock for the duration (so a
boot barrier can't latch the device into deep sleep mid-wait) and are safe from
any task.

A straddle that wants to defer activation until the `boot` script's
customisations are in subscribes to the boot-complete signal instead of polling:

```cpp
storageSubscribeChanges("sys.boot_complete", onBootComplete);
```

## Firmware identity

What the firmware *is* — its lowercase stub, display name, and banner — is fixed
at compile time from `CONFIG_SPANGAP_FW_STUB` / `_NAME` / `_BANNER` (set in
`straddle.yaml` as `stub:` / `display_name:` / `banner:`). It is exposed
read-only to the browser as the `fw.*` storage subtree and is never persisted or
settable. See the [storage](storage.md) doc for the `fw.*` surface. The
*mutable* hostname (what the device runs **on**) is the separate, user-editable
`s.net.hostname`, seeded from `CONFIG_SPANGAP_FW_HOSTNAME` (owned by spangap-net).

## Storage surface

init owns the project-identity and boot/build telemetry keys.

### Settings (`s.sys.*`)

| Key | Default | Meaning |
|---|---|---|
| `s.sys.project` | `CONFIG_SPANGAP_PROJECT_NAME` (compile-time, e.g. `spangap`) | Immutable project identity, written once on first boot. On every later boot `spangapInit()` compares the stored value to the compile-time one and **factory-resets `/state` + reboots** if they differ — so flashing a different spangap project over the same chip starts clean. Not user-editable. |
| `s.sys.time_wait_s` | `30` (inline fallback; not seeded) | Default budget for `waitForTime()`. `0` skips the wait on an offline node. |

### Telemetry (ephemeral, written at boot)

| Key | Meaning |
|---|---|
| `sys.boot_complete` | Set to `1` by `spangapPostAppInit()` after the boot script runs — the signal late-activation code subscribes to. |
| `sys.time.valid` | `1` once a time source has synced (published by the time source; read by `waitForTime`). |
| `sys.time.set` | Browser pushes epoch seconds here; the time source accepts it when the clock is invalid. |
| `sys.going_down` | Set when the last power lock releases — cron acts on it to enter deep sleep (see [power-management](power-management.md)). |
| `sys.build_time` | Compact `a<app> f<fixed> w<web>` build-epoch summary. |
| `sys.buildtime.{app,fixed,web}` | Build epochs: firmware link time, `fixed` source mtime, webroot CRC32. |
| `sys.build.{straddle,version,args}` | The `spangap build` invocation identity (build straddle, version, full argument string). |

`CONFIG_SPANGAP_SDCARD` gates whether `spangapInit()` mounts an SD card during
bring-up (see [fs](fs.md) for the state-store choice that follows).
