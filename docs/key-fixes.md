# Key Fixes (hard-won knowledge — spangap)

Bug fixes whose root cause is non-obvious and likely to be re-encountered or accidentally undone, scoped to platform code. App-domain fixes (camera, audio, recording, app-level protocols) live in the consumer's own docs.

## SD / FAT / filesystem

- **SD-card empty-slot noise**: `esp_vfs_fat_sdmmc_mount()` retries internally and IDF logs `sdmmc_init_ocr send_op_cond returned 0x107` / `vfs_fat_sdmmc: sdmmc_card_init failed (0x107)` on each attempt before returning. Fix: `esp_log_level_set("sdmmc_common"|"sdmmc_sd"|"vfs_fat_sdmmc", ESP_LOG_NONE)` around mount, restore `ESP_LOG_WARN` on success. Both probe + format-retry attempted unconditionally — a card sometimes only succeeds on the second probe; an empty slot fails both and we declare "no SD" with a single `warn()`.
- **FAT 8.3 filename limit**: strftime-generated filenames (e.g. `20260315_213549.avi`) exceed 8.3 chars → `fopen` returns "Invalid argument". Fix: `CONFIG_FATFS_LFN_HEAP=y` in sdkconfig.defaults.
- **PSRAM task stacks + SPI flash**: `xTaskCreatePinnedToCoreWithCaps(MALLOC_CAP_SPIRAM)` puts task stack in PSRAM. But SPI flash reads disable the PSRAM cache (shared bus), so the call stack must be in DRAM during flash ops. Fix: `fs.cpp` owns a DRAM-stack `fs` worker task that proxies all file I/O. `cliRunFile()` uses `fs_open`/`fs_read`/`fs_close` so it's safe from any PSRAM-stack task (e.g. `net_up` temp task). Web (LittleFS file serving) stays on DRAM.

## NTP / mDNS / network

- **NTP sync callback never fires**: `esp_sntp_set_time_sync_notification_cb()` callback never called despite time being set. Workaround: one-shot FreeRTOS timer 30 s after `ntpBegin()` to log time.
- **NTP server pointer**: `esp_sntp_setservername()` stores pointer, not copy — `server` buffer must be `static`.
- **mDNS/NTP on network down**: `mdns_free()` + `ntpStop()` before `esp_wifi_stop()` so goodbye multicast can be sent while RF is up. Also stops periodic timers that prevent sleep.
- **WiFi down 5 mA gap**: `esp_wifi_stop()` leaves driver tasks alive. Fix: `esp_wifi_deinit()` on network down + `esp_wifi_init()` on network up.
- **`sys_evt` stack overflow** (first STA connect / DHCP): default ESP event-loop task stack (2304) can overflow with esp_netif + WiFi + our log path. Fix: `CONFIG_ESP_SYSTEM_EVENT_TASK_STACK_SIZE=8192` in `sdkconfig.defaults`.
- **Network graceful shutdown**: `net down` clears `rtcWantUp`, waits for 30 s idle (no `netActivity()` calls from rtsp/web/log/cli). `net down!` forces immediate teardown. Consumer tasks call `netActivity()` in their active loops to signal network usage. `net up` sets `rtcWantUp` and cancels pending shutdown. PM `NO_DEEP_SLEEP` lock held from `net up` until fully down.
- **AP scan tears down AP clients**: `ST_AP` switched to `WIFI_MODE_STA` for scanning, dropping all AP clients. Fix: use `WIFI_MODE_APSTA` for non-disruptive scan, only tear down AP on successful STA connect.
- **WiFi AMPDU TX suspected of WebRTC packet loss** (under retest 2026-05): Block Ack teardown (DELBA reason:39) was observed alongside 5 s+ packet loss that killed DataChannel sessions. We previously disabled AMPDU TX (`CONFIG_ESP_WIFI_AMPDU_TX_ENABLED=n`), but the underlying ~6 min hang persisted, so the fix was suspect. Override removed on the IDF 5.5.4 upgrade — if the hang reappears with AMPDU TX on, restore the override and dig deeper into the actual root cause.

## WireGuard

- **WireGuard 4 KB stack overflow**: Curve25519 scalar multiply corrupted handshake state silently — server received bytes but couldn't validate. Fix: 8 KB stack for wg task.
- **WireGuard crypto on tcpip_thread**: per-packet ChaCha20-Poly1305 runs inline on tiT (~20% CPU at 14 fps). tiT has no competing workload, so offloading to a separate crypto task buys nothing and adds complexity.
- **WireGuard `netif->state` conflict**: ESP-IDF 5.x DHCP callback dereferences `netif->state` as `esp_netif`, but wireguard-lwip stores `wireguard_device` there. Fix: `CONFIG_LWIP_PPP_SUPPORT=y` switches to `netif_get_client_data()`. (Also the reason `esp_wireguard` is currently vendored under `src/esp_wireguard/` — once trombik's upstream propagates the workaround we drop the fork and depend on the registry component.)

## Power management / USB / sleep

- **USB SOF prevents light sleep**: even after releasing USB PM lock, USB SOF packets (1 ms) wake CPU. Fix: `usb down` disables D+ pullup via `usb_serial_jtag_ll_phy_enable_pull_override()` to signal disconnect to host.
- **USB Serial JTAG echo requires TX FIFO flush**: `printf()`/`fflush(stdout)` only pushes data into the USB FIFO; `usb_serial_jtag_ll_txfifo_flush()` is needed to actually send it out over USB without a newline trigger. Wrapped in `cliFlush()`.
- **USB PM lock released before host enumerates**: `pmPollUsb()` checks `usb_serial_jtag_is_connected()` (SOF detection) but SOF doesn't flow until host enumerates. If checked too early after boot, lock released → light sleep → USB clock gated → enumeration never completes. Fix: 5-second grace period keeps USB lock held after boot.
- **USB reconnect after light sleep**: after `usb down` + light sleep, restoring D+ pullup alone isn't enough — the USB Serial JTAG peripheral's internal state machine is confused. Fix: `usb up` does full peripheral reset via `usb_serial_jtag_ll_reset_register()` + re-enables internal PHY and pads.
- **USB state lost across deep sleep**: `usb down` only set a volatile flag — deep sleep wake re-acquired USB lock, preventing further deep sleep when host is connected. Fix: `rtcUsbDisabled` in RTC RAM. On deep sleep wake with USB disabled, `pmInit()` disables D+ pullup immediately and skips lock acquisition. Power cycle resets to normal.
- **Accidental deep sleep without cron**: `usb down` for light sleep testing could trigger deep sleep if no cron was configured, sending the device into unrecoverable deep sleep (no wake source). Fix: cron holds a `PM_NO_DEEP_SLEEP` lock by default, only releases when `s.cron.enable=1` AND crontab has entries.
- **PM profiling sscanf parse failure**: `esp_pm_impl_dump_stats` uses `%-3PRIu32` for MHz values, producing variable padding (e.g. `"80 M"` not `"80M"`). sscanf with literal `M` in format only matched `240M`. Fix: `%*[^M]M` scanset skips all chars up to `M`.
- **Server idle polling prevents DFS**: networking server tasks polled `select()` every 200 ms for incoming connections even when their server socket was closed (network down or port=0), keeping core 1 awake. Fix: block indefinitely on `ulTaskNotifyTake` when server socket is closed; wake on `net.up` subscription or config change.
- **`RTC_DATA_ATTR` does not survive `esp_restart`**: a factory-reset flag in `RTC_DATA_ATTR` silently re-initialized to 0 on `esp_restart()`, so the deferred format never fired on the next boot. Subtle ESP-IDF semantics:
  - `RTC_DATA_ATTR` is preserved **only** across deep-sleep wake. Re-initialized from its static initializer on every other reset (`esp_restart()`, watchdog, panic).
  - `RTC_NOINIT_ATTR` is preserved across **any** reset (warm reboot, esp_restart, panic, deep sleep). Contents are random on cold power-on.

  The `rtcAppMagic` + `rtcRamValid()` pattern works because deep-sleep wake is its only use case. Most existing `rtcUsb*`, `rtcPm*`, `rtcCkpt*` vars use `RTC_DATA_ATTR` and lose state on warm reboot, which is fine for "across deep sleep only" lifecycles.

  **Rule of thumb**: anything that needs to signal across a warm `esp_restart` *must* be `RTC_NOINIT_ATTR` and guarded by a 32-bit magic constant (magic matches → pending; anything else → clean state, which handles cold-boot randomness). For deep-sleep-only state, `RTC_DATA_ATTR` + `rtcRamValid()` is fine. Don't rely on `RTC_DATA_ATTR` to survive `esp_restart`, watchdog, or panic — it won't.

- **IDF 5.5+ USB Serial JTAG console dies in light sleep**: `usb_serial_jtag_vfs_read/write` ([components/esp_driver_usb_serial_jtag/src/usb_serial_jtag_vfs.c](file:///Users/rop/.espressif/v5.5.4/esp-idf/components/esp_driver_usb_serial_jtag/src/usb_serial_jtag_vfs.c)) now return `-1` immediately when `usb_serial_jtag_is_connected()` is false. That function tracks USB SOF packets — they stop arriving when light sleep gates the USB clock, flipping the connection state to false. Net effect: serial CLI input dies (reads return -1), logs may scrape through during bursty wake windows but eventually die too. Fix: enable `CONFIG_USJ_NO_AUTO_LS_ON_CONNECTION=y` in the consumer's `sdkconfig.defaults`. With it, IDF holds an internal `NO_LIGHT_SLEEP` lock from boot until SOFs *definitively* stop, breaking the death-spiral. Our own `pm.cpp` "usb" lock pre-dates this and is now redundant (still cheap, harmless).

## CLI / logging

- **CLI `[log]` prefix**: CLI responses (set/show/unset) must use `printf()` not `info()` — `info()` goes through log queue and adds `[task]` prefix.
- **Log task inbound bytes silently dropped (`DC→ITS drop: stream=N len=… target buffer full`)**: `itsServerPortOpen` was called with `toSize=0` (no client→server buffer). Fine when the server only sent; broken once the log task became bidirectional. Fix: `toSize=fromSize=2048` per connection.
- **Log paste-back exceeded buffer cap (`its: [log] packet (4+8128) exceeds buffer cap 2048`)**: `log:1` is packet-mode ITS — each `itsSend` is one atomic packet, header-prefixed. `logPasteBack`'s chunker capped at 8 KB, larger than the new 2 KB per-connection buffer. webrtc_task is built around 1-ITS-packet ↔ 1-DCEP-message (so switching to stream-mode would be a router-wide rework), and the steady-state line-per-message semantic is genuinely useful. Fix: drop chunk cap to 1500 in `logPasteBack` so the only oversize call (paste-back at DC connect) is split across multiple packets at `\n` boundaries.
- **Browser console messages didn't reach serial console**: serial isn't an ITS consumer of the log task — it sees output via `logVprintf`'s direct `fwrite(stdout)`. Inbound lines bypass `logVprintf`, so `logInboundLineOut()` must mirror the same direct stdout echo (gated by `serialInCli`).
- **Browser console hooks dropped early lines**: `dc.send` requires `readyState === 'open'`, but `console.log` calls during page boot fire before the WebRTC handshake completes. Fix: 128-slot bounded queue in `stores/log.ts` flushed on `dc.onopen`. Older queued lines are silently dropped if the DC never opens.
- **TCP CLI double echo**: line-mode clients (e.g. `nc`) show local echo, then server echoes each character via `cliEditChar`, then repeats the whole line on Enter. Fix: detect line-mode (recv returns >1 bytes ending with CR/LF) and process directly without char-by-char echo.
- **Log output overwrites CLI echo**: stream buffer drain (`cliDrainStream`) must be suppressed while `serialTyping > 0` (user is typing on serial), otherwise log output overwrites echoed characters.
- **CLI `serialTyping` must clear before `cliProcess`**: `serialTyping` (= `serialEdit.len`) stayed at command length during `cliProcess()` execution, causing log task to suppress stream writes for the entire command. Fix: reset `e.len`, `e.cursor`, and `serialTyping` to 0 before calling `cliProcess()` in `cliEditChar()`.
- **CLI output dropped on long commands**: `itsCliWrite` used `itsSend(handle, data, len, 0)` with timeout=0. The CLI→serial ITS buffer is 2048 bytes. Verbose commands like `pm` overflow the buffer; `itsSend` returns partial but the return was ignored — dropping the tail of the output, including the RESET escape and `$ ` prompt (terminal stuck in cyan, no prompt). Fix: retry loop with 100 ms timeout provides backpressure.
- **Task init port logging**: networking tasks (e.g. rtsp, web) open ports in their task function (not init function) so log messages show correct `[taskname]` tag instead of caller's tag.
- **FreeRTOS timer task stack overflow**: `localtime_r()` with DST timezone rules uses too much stack for the default 2048-byte Tmr Svc task. Avoid using `info()` or heavy formatting in timer callbacks.
- **Log stream buffer multi-writer corruption**: FreeRTOS stream buffers are single-writer. `logVprintf` called from any task — concurrent `info()` calls corrupted head/tail pointers (assert in `prvReadBytesFromBuffer`). Fix: `portMUX_TYPE` spinlock around `xStreamBufferSend` in `logVprintf`.
- **Log stream buffer in DRAM**: `logInputStream` allocated with `MALLOC_CAP_INTERNAL` — avoids PSRAM access from log task during operations that might interact with the cache/MMU. Task stack remains PSRAM.

## Storage / config

- **Config store use-after-free crash**: `storageSet`/`storageUnset` auto-commit path (`txDepth==0`) was unprotected — two tasks could race, both see `txDepth==0`, and one would `commitPatch` (merge + free `txPatch`) while the other was still building it. Corrupted `cfgRoot` → `cJSON_GetObjectItem` crash on freed node (`LoadProhibited` in `get_object_item`). Fix: recursive mutex wraps `storageBegin`/`storageEnd` (which own `txDepth`/`txPatch`), and all public readers lock around tree access. `storageSet`/`storageUnset` acquire the outer lock before checking `txDepth`.
- **`storage_change_msg_t` key truncation**: `key[28]` too small for `secrets.auth.cookies.N.expires`. Fix: `key[48]` + `val[44]`, `ITS_MAX_MSG_DATA` 64→96.
- **`stream.owner` stored as number**: all-digit random ID passed through `inferType` → stored as int → string comparison with `===` always false. Fix: prefix with `'c'` to force string type.
- **IPC_MAX_TASKS too small**: adding storage task as 13th registered task silently failed. Fix: bumped to 20.

## WebRTC / DataChannel / DTLS

- **DC reconnect loop — `stopStreaming` not called on re-offer**: `handleSignalingMsg` set `streaming = false` but never called `stopStreaming()`. Camera/audio subscribers leaked, hardware kept running. Fix: call `stopStreaming()` which properly unsubscribes, releases PM locks, stops hardware. (The hook is generic — application tasks register a teardown callback with the WebRTC router.)
- **DC reconnect loop — SCTP fragments silently dropped on WANT_WRITE**: `sctpSend` checked `ret < 0` but DTLS WANT_WRITE returned 0. The fragment was counted as sent but never went out. With WireGuard, EAGAIN on `sendto` caused ALL fragments of every frame to be silently lost — browser received nothing, hit 10 s signal timeout, reconnected. Fix: retry once after `vTaskDelay(1)` on WANT_WRITE; return partial count on persistent failure.
- **DC task watchdog — `vTaskDelay(1)` required**: DC task runs at priority 2 on core 0. Without yielding, IDLE0 (watchdog feeder) starves. WireGuard crypto on tiT can also hold lwIP mutexes, causing `recvfrom` to block and compound the starvation. `vTaskDelay(1)` per iteration is mandatory.
- **DC signaling WS disconnect didn't stop streaming**: `webrtcItsDisconnect` set `itsHandle = -1` but never called `stopStreaming()`. When browser closed, UDP streaming continued forever — application subscribers stayed registered, DTLS context leaked ~10 KB DRAM. Fix: full session teardown in disconnect callback (stopStreaming + dtlsSessionFree + SCTP reset).
- **WebRTC streaming continues after browser closes**: signaling WS disconnect triggered `stopStreaming()`, but if WS close frame never arrives (browser crash, network loss), UDP streaming continues forever. Fix: 10 s UDP inactivity timeout in webrtc main loop tears down session. Also: TCP keepalive on web endpoints (10 s idle, 5 s interval, 3 probes) detects dead connections within ~25 s. Browser `beforeunload` handler sends clean WS close for instant teardown.
- **DTLS renegotiation stalls streaming after ~5–10 min**: mbedTLS has a bug ([#687](https://github.com/Mbed-TLS/mbedtls/issues/687)) where the epoch field isn't masked in the record counter comparison, potentially triggering spurious renegotiation after ~65 K records. Fix: `mbedtls_ssl_conf_renegotiation(&dtlsConf, MBEDTLS_SSL_RENEGOTIATION_DISABLED)` — not needed for DataChannel anyway.
- **SLOW LOOP false alarm with WS polling**: WS clients use 200 ms poll timeout, slow loop detector threshold was also 200 ms. Fix: skip slow loop warning when `sessionPaused`.

## Web / auth / browser shell

- **Web endpoint residual buffer shared**: single global `epResidual` buffer for partial TCP writes was shared across all ITS endpoints. A partial write left data that got flushed to the next ITS endpoint's TCP client as binary noise. Fix: per-endpoint residual buffer in `ws_endpoint_t`.
- **Web task stack overflow from auth SHA-256**: `authPasswd` → `storageSave()` → `cJSON_Print()` chain + `mbedtls_sha256_context` overflowed 6 KB DRAM stack. Fix: removed `storageSave()` from `authPasswd` (save timer handles it on storage task's stack), bumped web task to 8 KB.
- **Chrome blocks password save on self-signed cert**: `chrome_password_manager_client.cc` checks `IsCertStatusError(GetMainFrameCertStatus())` — any cert error disables save prompt. Works on HTTP and with ACME certs. Form DOM removal before `router.push()` triggers Chrome's `CanInferFormSubmitted()` (requires valid cert).
- **FloatingWindow dock state lost on close→reopen**: the visibility watcher calls `undockWindow` on hide so a closed window doesn't reserve dock space, but the debounced `saveState` (500 ms `setTimeout`) read `docks[id].side` AFTER the undock had already nulled it, persisting `dock: null` and losing the user's docked layout across reload. Fix: snapshot dock side+size synchronously at `saveState` call time, then write the snapshot from the timer. Watch order matters too — the persistence-watcher is registered before the undock-watcher so the snapshot sees the docked state.
- **Resize handles clipped by `.fw overflow:hidden`**: outside-of-border hit zone was unclickable because the parent's `overflow: hidden` clipped pointer hit-testing alongside rendering. Fix: drop `overflow: hidden` on `.fw`, give `.fw-titlebar` and `.fw-body` their own `border-radius` to keep the rounded look, and lift `.fw-titlebar` to `z-index: 11` so close/drag still win inside the title-bar area while resize handles win along the body edges.

## ESP-IDF version regressions

- **IDF 5.5 `HEAP_TASK_TRACKING` IWDT crash on `cJSON_Delete`**: see [idf-tweaks.md](idf-tweaks.md). 5.5 rewrote heap tracking from on-demand walk (free) to eager bookkeeping under a global mutex (expensive, contention causes IWDT timeout). Bypass via `--wrap` link options + no-op stubs in `src/heap_track_stub.c`; guard in `CMakeLists.txt` fails the build if the wrapped function names ever change.
